Overview

FoodHub is a cloud-native, smart restaurant management platform designed to transition traditional dine-in and food delivery systems into a single high-performance enterprise ecosystem.

  • Who it is for: Mid-to-large scale restaurant franchises needing centralized control over reservations, waiter assignments, inventory, billing, KDS display boards, and real-time sales reporting.
  • Why it was built: To address latency in order placement, inventory slippage, unoptimized floor layouts, and lack of customer personalization in standard restaurant setups.
  • Core Objective: Harness event-driven serverless architectures to provide sub-second order processing alongside dynamic AI recommendations and business insights.

High-Level Architecture Model

The platform leverages a hybrid operational model. Relational transactional schemas reside in high-performance MySQL databases, while order delivery streams flow asynchronously through AWS SQS queues into decoupled Node.js processors and AWS DynamoDB tables to ensure linear write scalability.

Additionally, Gemini generative models are embedded to analyze post-dining feedback sentiment, offer intelligent menu recommendations, and predict peak-hour customer demand charts for administrators.

The Problem

Traditional restaurant frameworks suffer from isolated operational layers:

  • Ordering Bottlenecks: Peak load times cause request queuing, leading to system drops or delayed confirmations.
  • Inventory Blind Spots: Lack of real-time ingredient deductions against incoming orders leads to stockouts or waste.
  • Unstructured Feedback: General ratings fail to highlight specific issues in food quality, service delays, or ambient complaints.
  • Operational Disconnection: Cashiers, chefs, and waiters operate on disjointed updates, causing order delivery mismatches.

The Solution

FoodHub unifies dining and supply chains using cloud patterns:

  • Asynchronous SQS Queuing: Decouples customer requests from database processes, mitigating spikes in load.
  • Real-time Stock Tracking: Automated deductions and low-stock alerts notify managers instantly during checkout.
  • Gemini Sentiment Analysis: Parses textual feedback to automatically categorise positive or negative themes.
  • Centralized KDS and Waiter Dashboards: Displays live orders on kitchen screens and automates waiter requests instantly.

AWS Infrastructure Footprint

Leveraging Amazon Web Services to build an enterprise-ready infrastructure featuring high scalability and security.

EC2 Compute

Hosts the core Node.js application server behind an Nginx reverse proxy. Managed with auto-restart daemons to guarantee service runtime.

Lambda Serverless

Runs asynchronous workers triggered directly by SQS message queues to process customer checkout states without using EC2 resources.

RDS (MySQL) Database

Provides highly available transactional storage. Isolated in private subnets with access restricted via strict Security Group rules.

SQS Message Queue

Asynchronously buffers incoming transaction requests to protect back-end databases from high-concurrency spikes.

S3 Object Storage

Serves dynamic assets such as menu item images and generated invoice PDFs, using automated lifecycle rules to reduce costs.

IAM Access Management

Implements granular access policies, ensuring EC2 instances and Lambda functions only hold the minimum permissions required to write to SQS or DynamoDB.

Engineering Challenges

How specific performance, configuration, and API limitations were resolved during development.

SQL mode=only_full_group_by

Problem

Queries to fetch popular dishes and active waiter tables failed under default SQL strict group-by modes.

Solution

Re-structured queries to aggregate non-grouped columns and explicitly grouped by functional dependencies.

Outcome

Eliminated query drops and achieved full MySQL 8.0 strict mode compliance.

Local and Cloud DB Parity

Problem

Non-standard SQL migrations failed on MySQL 8.0 instances due to "IF NOT EXISTS" alter column syntax constraints.

Solution

Re-authored migrations using standard SQL syntax statements and wrote connection probes to verify DB states.

Outcome

Achieved cross-environment schema consistency across local dev servers and AWS RDS instances.

Gemini API Key Expiry

Problem

Network latency or invalid API keys would halt core recommendation and checkout routes.

Solution

Developed dynamic error interception middleware providing deterministic fallback lists during API downtime.

Outcome

Ensured 100% route uptime even during third-party API outages.
